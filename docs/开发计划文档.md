# 隐私拨号 App 开发计划文档

## 1. 文档目标

本文档基于 [方案书](./方案书.md) 与 [设计文档](./设计文档.md) 输出可执行的开发计划，用于指导项目立项、排期、分工、联调、测试与上线准备。

本计划默认面向首发版本 `V1.0`，目标是完成“登录后使用的隐私拨号 App”首发交付，并保留增强版与运营版的后续演进路径。

## 2. 计划前提

### 2.1 项目定位

- 产品定位为“尝试隐藏来电显示，从而间接隐藏号码归属地”。
- 不承诺 `100%` 隐藏真实号码或地区信息。
- 实际效果取决于运营商、网络制式、SIM 卡、系统拨号器与终端环境。
- 首发版本以中国大陆场景优先，默认前缀为 `#31#`。

### 2.2 排期假设

本计划按一个精简团队估算，总周期建议为 `10 周`：

| 角色 | 建议投入 |
|------|----------|
| 产品/项目经理 | 1 人 |
| Flutter 开发 | 1 人 |
| NestJS 后端开发 | 1 人 |
| 测试工程师 | 1 人 |
| 运维/发布支持 | 0.5 人 |

如增加 1 名客户端或后端开发，整体周期可压缩约 `1-2 周`；如测试资源不足，真机兼容阶段将明显拉长。

## 3. 版本范围

### 3.1 V1.0 必做范围（P0）

| 模块 | 交付内容 |
|------|----------|
| 登录鉴权 | 手机号验证码登录、`send-code/login/refresh/logout/me` 接口、JWT 鉴权、登录态守卫 |
| 拨号核心 | 普通拨打、隐私拨打、号码清洗校验、前缀拼接、系统拨号器拉起 |
| 配置能力 | 前缀配置接口、公告接口、客户端默认前缀读取 |
| 页面交互 | 启动页、登录页、验证码页、主页、设置页、帮助页、关于页 |
| 安全风控 | Token 安全存储、验证码哈希存储、手机号/IP 限流、日志脱敏 |
| 测试验收 | 单元测试、基础集成测试、服务端 E2E、运营商/机型兼容矩阵 |
| 上线基础设施 | Docker 化部署、基础 CI/CD、错误监控、环境配置管理 |

### 3.2 V1.1 建议纳入首发缓冲的增强项（P1）

| 模块 | 交付内容 |
|------|----------|
| 客户端体验 | 最近号码本地记录、首次风险提示、Token 自动刷新与 401 重试 |
| 服务端能力 | 后台动态前缀管理、公告管理、风控阈值可配置 |
| 质量体系 | Flutter `analyze/test`、NestJS 单测与 E2E 进入 CI |

### 3.3 上线后版本（P2）

| 模块 | 交付内容 |
|------|----------|
| 运营能力 | 管理后台、数据看板、用户反馈闭环 |
| 登录扩展 | Apple 登录、Google 登录、更多登录方式 |
| 全球化 | 更多国家/地区前缀、运营商精细化推荐 |
| 发布运营 | 应用商店持续迭代、灰度与召回策略完善 |

## 4. 里程碑计划

### 4.1 总体阶段

| 阶段 | 周期 | 阶段目标 | 退出标准 |
|------|------|----------|----------|
| 阶段 0：项目启动 | 第 1 周 | 冻结首发范围，完成环境、仓库、接口与数据模型对齐 | 技术方案冻结、环境可运行、任务拆分完成 |
| 阶段 1：核心链路开发 | 第 2-4 周 | 跑通登录、鉴权、配置、基础页面与服务端主链路 | 登录链路联调通过，数据库与核心 API 可用 |
| 阶段 2：拨号能力开发 | 第 5-6 周 | 完成普通拨打、隐私拨打、前缀选择、设置/帮助页 | 客户端可完成完整拨号流程，关键异常有提示与回退 |
| 阶段 3：联调与验证 | 第 7-8 周 | 完成客户端/服务端联调、自动化测试、兼容矩阵验证 | P0 缺陷收敛，兼容结果可评估是否上线 |
| 阶段 4：上线准备 | 第 9-10 周 | 完成部署、监控、发布材料、验收与灰度上线准备 | 满足上线门槛，具备首发条件 |

### 4.2 周计划

| 周次 | 重点目标 | 客户端重点 | 服务端重点 | 测试/运维重点 |
|------|----------|------------|------------|---------------|
| 第 1 周 | 项目启动 | Flutter 项目骨架、路由骨架、依赖清单 | NestJS 骨架、数据库方案确认、环境变量模板 | 测试计划初版、部署环境准备 |
| 第 2 周 | 鉴权主链路 | 登录页/验证码页、协议勾选、Secure Storage | `send-code/login`、短信接入、验证码表与用户表 | 接口测试用例、短信链路联调 |
| 第 3 周 | 登录态闭环 | Splash、路由守卫、登录态恢复 | `refresh/logout/me`、Refresh Token 轮换、JWT 守卫 | 鉴权回归、异常码校验 |
| 第 4 周 | 配置与基础页面 | 主页骨架、设置页、帮助页、公告展示 | `dial-prefixes/notices`、前缀配置表、公告表 | 页面冒烟、配置联调 |
| 第 5 周 | 拨号能力实现 | 号码校验、普通拨打、隐私拨打、错误提示 | 风控限流、配置缓存优化 | 拨号流程测试、异常回退测试 |
| 第 6 周 | MVP 功能封板 | 最近号码、前缀切换、首次风险提示 | 日志脱敏、监控埋点、接口补齐 | MVP 回归、缺陷收敛第一轮 |
| 第 7 周 | 自动化与兼容 | 集成测试、真机问题修复 | 单测/E2E、容器化联调 | 机型矩阵、运营商矩阵验证 |
| 第 8 周 | 质量封板 | UI/交互修正、发布包验证 | Docker Compose、CI/CD、告警接入 | 全量回归、发布候选版验收 |
| 第 9 周 | 上线准备 | 商店素材、版本配置、灰度开关 | 生产环境部署、Secret 配置、监控看板 | 预生产验证、上线 Checklist |
| 第 10 周 | 首发验收 | 发布支持、问题修复 | 灰度观察、接口稳定性保障 | 上线验收报告、首周观察机制 |

## 6. 关键依赖与决策点

| 决策项 | 最晚确认时间 | 影响 |
|--------|--------------|------|
| 短信供应商选型（阿里云或腾讯云） | 第 1 周结束前 | 影响验证码联调、成本与可达性 |
| JWT 签名方案（HS256 或 RS256） | 第 2 周结束前 | 影响密钥管理与部署复杂度 |
| 首发是否包含最近号码与公告系统 | 第 3 周结束前 | 影响客户端与服务端工作量 |
| 真机与 SIM 卡测试资源清单 | 第 4 周结束前 | 影响兼容矩阵测试是否能如期完成 |
| 首发是否带后台管理入口 | 第 6 周结束前 | 影响是否扩展为 P1 或延后到 P2 |
| 监控方案（Sentry/Crashlytics/服务端告警） | 第 7 周结束前 | 影响上线观察与问题定位效率 |

## 7. 风险与应对

| 风险 | 等级 | 表现 | 应对措施 |
|------|------|------|----------|
| 运营商不支持 `#31#` | 高 | 隐私拨打无效，用户看到号码 | 上线前完成运营商矩阵验证，保留普通拨打回退，帮助页明确说明 |
| 部分系统拨号器对 `#` 处理异常 | 高 | 前缀被裁剪或拨号失败 | 优先真机验证主流机型，保留可切换前缀，记录失败机型清单 |
| 用户误解产品能力 | 高 | 误认为可 100% 隐藏来电信息 | 在登录、帮助、风险提示、商店文案中统一能力边界 |
| 短信可达性与成本波动 | 中 | 登录失败率上升，成本超预期 | 预留双供应商切换能力，做发送限流与监控 |
| iOS/Android 平台差异过大 | 中 | 双端行为不一致 | 使用统一测试脚本，分别维护平台差异说明 |
| 测试资源不足 | 中 | 兼容矩阵不完整，影响上线判断 | 第 1 周即锁定真机和 SIM 资源，必要时分批灰度 |

## 8. 上线门槛

以下条件全部满足后，才建议进入首发上线：

- 登录主流程稳定可用，未登录不可进入拨号主页面。
- 普通拨打与隐私拨打均可正常拉起系统拨号器。
- Token 刷新、退出登录、异常回退均经过回归验证。
- 三大运营商至少完成一轮真机验证，并形成兼容结论。
- 服务端具备限流、脱敏、监控、日志与基础告警能力。
- 帮助页、协议页、商店文案已明确能力边界，不存在误导描述。
- 预生产环境完成部署演练，具备回滚方案。

## 10. 推荐执行策略

建议采用“`P0 先闭环、P1 控制范围、P2 延后运营化`”的推进方式：

- 第一个目标不是“功能多”，而是稳定跑通“登录 → 配置 → 隐私拨打 → 回退”主路径。
- 真机兼容验证应与开发并行推进，不能等开发完成后再集中发现运营商问题。
- 首发版本不建议同时追求管理后台、多登录方式和全球化前缀支持，以避免核心链路被稀释。
- 如果第 7-8 周兼容矩阵显示中国电信或 iOS 存在明显不稳定，应及时调整首发文案、支持范围或灰度策略，而不是强行全量上线。

以上计划可直接作为 `V1.0` 项目排期基线；若团队规模、预算或首发范围发生变化，可优先调整第 3 章版本范围与第 4 章里程碑周期。
