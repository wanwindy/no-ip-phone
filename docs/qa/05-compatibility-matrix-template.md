# Round 3 真机兼容矩阵（执行版）

## 1. 用途

本文件是 Round 3 的真机执行资产，不再只是占位模板。它用于记录真实设备、真实 SIM、真实网络环境下的隐私拨打表现，帮助后续把“能不能拨通、能不能拉起拨号器、被叫侧看到什么”拆开验证。

重要约束：

1. 只记录真实实测结果，不填推测值。
2. 不要用单台设备的表现推导全部机型。
3. 如果结果无法确认，直接写“待确认”并补证据，不要空着。
4. 如需引用 Round 2 联调结论，只保留已确认契约，`logout` 仍表示“注销当前 refresh token”，不要把它写成 access token 立即失效。

## 2. 最低覆盖建议

| 维度 | 最低覆盖 |
|------|----------|
| 平台 | Android / iOS |
| 机型 | 至少覆盖主流品牌各 1 台，优先补拨号器差异明显的设备 |
| 系统版本 | 选择当前可用的主流版本，优先新旧各一档 |
| 运营商 | 中国移动、中国联通、中国电信 |
| 网络制式 | 4G / VoLTE / 5G；如有条件补测弱网或关闭 VoLTE 场景 |
| 前缀 | 默认 `#31#`，以及 Round 3 冻结的其他候选前缀 |
| 拨号方式 | 手动输入、联系人、最近号码、复制粘贴等真实入口 |
| 场景 | 普通拨号、隐私拨号、失败回退、帮助页校验 |

## 6. 结论分级

| 结论 | 含义 | 处理 |
|------|------|------|
| 通过 | 结果符合预期 | 记录为可用组合 |
| 部分通过 | 普通拨号可用，但隐私拨号或被叫显示存在差异 | 标注限制条件，纳入帮助说明 |
| 不通过 | 拨号失败、拨号器无法拉起，或行为与预期明显不符 | 进入缺陷或阻塞项 |
| 待确认 | 现阶段证据不足，无法判断 | 补录截图、录屏或日志后再定性 |

## 7. 填报要求

1. 每个“异常”或“待确认”组合必须附截图或录屏。
2. 不能确认被叫显示结果时，必须写明“为什么无法确认”。
3. 每个“通过”的隐私拨打组合，必须写清运营商、网络制式和前缀。
4. 如果同一设备出现“能拉起拨号器但后续取消”的情况，也要单独记录，不要和成功拨出混写。
