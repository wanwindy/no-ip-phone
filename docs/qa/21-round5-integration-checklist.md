# Round 5 联调清单

> 日期：2026-04-22  
> 适用轮次：Round 5 / 第一轮  
> 目标：为 A/B/C/D 四个线程提供第一轮可执行联调顺序、准备项、失败分流和证据模板。

## 1. 联调目标

- 用最小数据集打通多租户、出呼任务壳、回拨命中规则、双模式拨号抽象和契约一致性。
- 在不引入真实 trunk 互联的前提下，先完成“数据 + API + 状态 + 文档”联调闭环。
- 为下一轮继续接入真实编排链路提供统一门禁入口。

## 2. 联调参与方与输入物

| 线程 | 必备输入物 | 联调关注点 |
|------|------------|------------|
| A | migration、entity、建模说明 | `tenant_id`、DID 分配、`call_session`、`callback_session` |

| C | webhook / event / call-state / callback 规则文档 | 字段一致性、状态字典、幂等键、TTL |
| D | 测试追踪矩阵、冒烟/回归模板、门禁模板 | 执行顺序、证据归档、放行结论 |

## 5. 联调执行顺序

1. 先校验 A 线程输出是否具备最小可用数据模型和租户隔离字段。
2. 再校验 C 线程契约文档是否为 B 线程 API 壳和状态占位提供唯一口径。
3. 再联通 B 线程 API 壳和客户端双模式抽象，确认两条分支可区分。
4. 最后执行跨线程场景：多租户隔离、DID 外显、同号回拨、双模式切换、契约变更回归。

## 7. 失败分流规则

- 命中租户隔离问题，优先回 A 线程排查数据模型和租户上下文，再由 B 线程确认接口透传。
- 命中 `POST /calls/outbound` 字段不一致或缺字段，优先回 B 线程，必要时同步 C 线程修正文档口径。
- 命中 callback 规则不一致，优先回 C 线程确认规则冻结，再回 A 线程核查数据落点。
- 命中双模式切换错误，优先回 B 线程，不允许用“暂时默认旧模式”掩盖问题。
- 命中无差异记录的契约变化，直接升级主线程评审，不允许继续联调。
