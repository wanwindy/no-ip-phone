# Round 2 联调执行指南

## 1. 目标

Round 2 只做 `auth/config` 联调闭环，目标是验证客户端、服务端和 QA 资产能按统一顺序完成登录与配置主链路，不展开正式发布动作。

## 2. 联调顺序

推荐执行顺序固定为：

1. `POST /api/v1/auth/send-code`
2. `POST /api/v1/auth/login`
3. `GET /api/v1/auth/me`
4. `GET /api/v1/config/dial-prefixes`
5. `GET /api/v1/config/notices`
6. `POST /api/v1/auth/refresh`
7. `POST /api/v1/auth/logout`

## 3. 执行原则

- 先验证固定验证码策略，再验证 token 与配置接口。
- 先验证成功路径，再验证错误路径。
- 先验证登录态，再验证受保护接口。
- 如果限流未完全落地，相关断言只记录为“待验证/受实现影响”，不强行判通过。
- 不在 Round 2 进入拨号真机效果评估，也不做发布签核。

## 4. 逐步执行说明

### 4.1 `send-code`

确认联调环境满足非生产、`SMS_PROVIDER=noop`、`AUTH_FIXED_CODE=123456`。调用后只检查响应 `cooldown`、状态码和日志，不寻找调试验证码字段。

### 4.2 `login`

使用同一手机号和固定验证码 `123456` 登录，检查是否返回 `accessToken`、`refreshToken`、`expiresIn`，并确认客户端登录态可持久化。

### 4.3 `me`

在登录成功后立即调用，确认返回的用户信息与脱敏手机号格式符合预期。

### 4.4 `dial-prefixes`

确认能拿到 CN 默认前缀与预置运营商前缀，重点关注返回结构、鉴权、排序和兜底前缀。

### 4.5 `notices`

确认公告接口可返回稳定数据。若当前无公告内容，允许返回空数组，但响应结构必须稳定。

### 4.6 `refresh`

用当前 `refreshToken` 刷新 token，检查是否轮换成功。若限流相关实现仍未落地，不把该接口与验证码限流混为一谈。

### 4.7 `logout`

退出后优先复查原 `refreshToken` 是否已被注销。当前 Round 2 契约不要求已签发 access token 立即失效，因此如果 `me` 仍可访问，应记录为“符合当前契约”，而不是直接判失败。

## 5. 结果记录要求

每一步都要记录：

- 实际请求时间
- 请求参数
- HTTP 状态码
- 业务响应码
- 实际返回数据摘要
- 与预期的差异
- 责任归属

## 6. 失败处理

1. 先判断是环境问题、数据问题还是实现问题。
2. 如果是固定验证码策略未生效，先核对环境变量。
3. 如果是接口契约不一致，停止联调并回报主 agent。
4. 如果是限流未落地，按“待验证/受实现影响”记录，不误判通过。
