# Round 4 真机最小覆盖计划

## 1. 目标

Round 4 的真机验证目标不是“尽可能多测”，而是在发布前用最小但足够的样本，回答下面四个问题：

1. 候选版本是否在 Android / iOS 主路径上可用。
2. 中国移动 / 中国联通 / 中国电信 是否至少各有一组真实验证结果。
3. 隐私拨打、普通拨打回退、异常回退、帮助文案 是否已经形成可放行结论。
4. 未覆盖项是否已经被明确标记为限制、风险或阻塞，而不是默认视为通过。

## 2. 使用边界

1. 本计划只定义 Round 4 的最小放行覆盖，不等于完整兼容性承诺。
2. 没有真实设备结果的组合只能写“未执行”或“待确认”，不能写“通过”。
3. 本计划默认复用 [05-compatibility-matrix-template.md](./05-compatibility-matrix-template.md) 记录单条组合结果，复用 [14-round3-dial-exception-fallback-checklist.md](./14-round3-dial-exception-fallback-checklist.md) 记录异常回退。
4. 如果现场设备资源不足，先保证“必测项”完整，再扩展“补充项”。

## 3. 进入执行前的准入条件

以下条件未满足时，不建议进入真机执行，也不应给出放行建议：

| 准入项 | 要求 | 责任归属 |
|--------|------|----------|
| 候选包冻结 | 版本号、构建号、变更范围明确，执行期间不再换包 | 主线程 + Client Worker |
| 预生产环境可用 | 登录、配置、验证码、日志定位入口可用 | Server Worker + 运维 |
| 设备池锁定 | 至少锁定本计划中的 Android / iOS 最小设备池 | QA / Release |
| SIM/号码池锁定 | 至少准备 3 个运营商的真实 SIM 或等效验证资源 | QA / Release |
| 证据路径准备 | 截图、录屏、日志、日报保存路径已建好 | QA / Release |
| 回填入口明确 | 兼容矩阵、问题池、日报模板已指定负责人 | QA / Release |

## 4. 最小放行样本

### 4.1 平台与设备池要求

| 样本池 | 最低数量 | 选择规则 | 说明 |
|--------|----------|----------|------|
| Android 主力机 | 2 台 | 选择国内主流厂商、不同系统拨号器实现 | 优先华为 / 小米 / OPPO / vivo 中差异明显机型 |
| Android 补充分层机 | 1 台 | 选择与主力机不同的系统版本档位 | 优先覆盖较旧 Android 主版本 |
| Android 风险复测机 | 1 台 | 用于复测异常、边界机型或高风险厂商 ROM | 没有高风险线索时，可从现有 Android 池中复用 |
| iPhone 当前主版本 | 1 台 | 当前主流 iOS 大版本 | 必须纳入最小放行样本 |
| iPhone 次主版本 | 1 台 | 比当前版本低一档的 iOS 大版本 | 没有独立设备时，需明确写成阻塞或限制 |

最小设备池合计：`Android 4 台 + iPhone 2 台`。

### 4.2 运营商与网络要求

| 维度 | 最低要求 | 备注 |
|------|----------|------|
| 运营商 | 中国移动 / 中国联通 / 中国电信 各至少 1 组真实结果 | 结果可以是通过、受限或不通过，但不能缺失 |
| 网络制式 | 4G / 5G 至少各覆盖 1 组；如现场具备条件，补测 VoLTE | 若无法控制网络制式，需在备注说明现场实际状态 |
| 被叫侧观察 | 至少 1 个稳定被叫观察号码 | 需要可回看被叫显示结果 |

### 4.3 必测路径

下表中的必测路径是 Round 4 的最小放行集合。每一项都要有真实结论和证据编号。

| 编号 | 路径 | 最低覆盖要求 | 通过标准 | 不通过处理 |
|------|------|--------------|----------|------------|
| R4-RD-01 | 普通拨号基线 | Android / iOS 各至少 1 台 | 不加前缀可拉起系统拨号器 | 归类为主链路阻塞候选 |
| R4-RD-02 | 隐私拨号基线 | Android / iOS 各至少 1 台 | 加冻结前缀后可拉起拨号器，并记录被叫显示 | 进入缺陷池或能力限制判断 |
| R4-RD-03 | 手动输入 | 至少 2 台 Android + 1 台 iPhone | 可正常完成输入并发起拨号 | 记录具体输入路径与提示 |
| R4-RD-04 | 联系人入口 | 至少 1 台 Android + 1 台 iPhone | 联系人路径可走通 | 如系统限制，单独标注为限制 |
| R4-RD-05 | 最近号码入口 | 至少 1 台 Android + 1 台 iPhone | 最近号码可回填并发起拨号 | 进入客户端缺陷或回退问题 |
| R4-RD-06 | 复制粘贴入口 | 至少 1 台 Android + 1 台 iPhone | 粘贴号码后可正常发起拨号 | 记录输入框、前缀拼接表现 |
| R4-RD-07 | 用户取消回退 | 至少 1 台 Android + 1 台 iPhone | 取消后返回状态符合预期 | 不得把用户取消记成系统故障 |
| R4-RD-08 | 非法号码回退 | 至少 1 台 Android + 1 台 iPhone | 有明确提示或阻止继续 | 无提示或误提示进入缺陷池 |
| R4-RD-09 | 帮助/提示一致性 | 候选包一次全量检查 | 帮助页、提示文案、能力边界一致 | 文案误导按高优先级问题处理 |

## 5. 执行顺序

建议按下面顺序做，避免在真机现场反复返工：

1. 先跑 `R4-RD-01` 普通拨号基线，确认不是设备本身或 SIM 本身不可拨。
2. 再跑 `R4-RD-02` 隐私拨号基线，确认候选前缀在当前设备/运营商上的表现。
3. 再补齐 `R4-RD-03` 到 `R4-RD-06` 的入口覆盖，优先完成最小样本。
4. 最后执行 `R4-RD-07` 到 `R4-RD-09` 的回退与文案检查。
5. 每完成一台设备，立即回填兼容矩阵和问题池，不要等到当天结束再补录。

## 6. 通过、受限与阻塞判定

| 判定 | 含义 | 是否可继续推进 |
|------|------|----------------|
| 通过 | 主链路符合预期，证据齐全 | 是 |
| 受限 | 行为可解释，有明确边界和替代路径 | 是，但必须进入风险台账 |
| 不通过 | 主路径失败、提示错误或证据显示行为明显异常 | 否，进入缺陷分级 |
| 待确认 | 现场证据不足，无法定性 | 否，补证据后再判定 |
| 未执行 | 因设备、SIM、环境未就绪未完成 | 否，默认视为放行阻塞项 |

## 7. 最小输出物

本计划执行完成后，至少输出以下四类材料：

1. 已回填的 [05-compatibility-matrix-template.md](./05-compatibility-matrix-template.md)。
2. 已回填的 [14-round3-dial-exception-fallback-checklist.md](./14-round3-dial-exception-fallback-checklist.md)。
3. 一份 Round 4 预发布 smoke 结论。
4. 一份 Round 4 问题收敛日报或结论。

## 8. 阻塞清单

以下任一项成立时，Round 4 真机验证只能输出“未具备放行条件”，不能输出“建议发布”：

1. Android / iOS 任何一侧没有真实设备结果。
2. 三大运营商中任一运营商完全没有结果。
3. 必测路径中存在“未执行”或“待确认”且未明确责任人与补测时间。
4. 证据缺失，导致“通过/受限/不通过”结论不可回溯。
5. 现场执行发现主链路异常，但没有进入缺陷池和收敛流程。
