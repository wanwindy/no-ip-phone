# Round 4 缺陷分级与放行门槛

## 1. 目的

Round 4 已进入“真机验证与预发布收口”阶段，缺陷分级不再只服务于记录，还直接决定：

1. 是否继续执行候选包验证。
2. 是否允许进入预发布 smoke。
3. 是否允许给出“建议发布”结论。

## 2. 分级原则

1. 先看是否影响首发主链路，再看影响面和是否有稳定替代路径。
2. 真机问题按“真实用户视角”定级，不按单条日志严重程度定级。
3. 文案误导、能力边界误报、错误结论错误，同样可以定为高等级。
4. 未执行、待确认、证据不足 不是低风险状态，不能自动降为 P2/P3。

## 3. 缺陷等级定义

| 等级 | 判定标准 | 典型示例 | 放行影响 |
|------|----------|----------|----------|
| P0 | 候选版本不可发布，已影响首发主链路或放行基础 | 无法登录、无法拉起拨号器、候选包不可安装、预发布 smoke 主链路失败、严重误导用户“保证隐藏成功” | 立即阻塞发布 |
| P1 | 主链路可部分执行，但在主流平台/主流运营商/最小覆盖样本上存在高概率失败或高投诉风险 | Android 或 iOS 主样本隐私拨号持续失败、三大运营商之一主链路不稳定、帮助/提示与真实行为明显冲突 | 默认阻塞发布 |
| P2 | 有明确替代路径或影响面可控，不阻断最小放行样本闭环 | 个别入口失败但手动输入可用、少量机型显示差异、日志提示不完整但不影响执行 | 可带病收敛后放行 |
| P3 | 低风险体验问题，不影响放行结论 | 轻微文案、排版、记录性问题 | 不阻塞发布 |

## 4. 定级辅助规则

| 维度 | 提升等级条件 |
|------|--------------|
| 平台 | 同一问题覆盖 Android / iOS 双平台时，至少提升一级评估 |
| 运营商 | 同一问题在三大运营商中复现两家及以上时，至少按 P1 评估 |
| 证据 | 有录屏、日志、被叫侧截图形成完整闭环时，优先按实证定级 |
| 文案 | 与真实能力边界相冲突的误导文案，最低按 P1 处理 |
| 发布窗口 | 发生在候选版本冻结后且无验证余量的问题，默认向高一级收敛 |

## 5. Round 4 放行门槛

### 5.1 硬门槛

以下条件任一不满足，都不能给出“建议发布”：

1. `P0 = 0`。
2. 最小真机覆盖计划已执行完毕，且不存在“未执行”或“待确认”的必测项。
3. 预发布 smoke 的阻塞项全部通过。
4. 生产 / 预生产环境已配置真实短信 provider，且不存在 `SMS_PROVIDER=noop` 或“仅日志输出验证码”的做法。
5. 回滚条件、回滚负责人、证据路径明确。
6. 文案、帮助、风险提示没有把“尝试隐藏来电显示”写成“保证成功”。

### 5.2 P1 门槛

Round 4 默认不带 `P1` 发布。只有同时满足以下条件时，才能讨论“限制范围放行”：

1. 问题不落在最小放行样本的必测主链路上。
2. 影响范围已经明确到具体平台 / 机型 / 运营商组合。
3. 已有可执行替代路径，且帮助文案、公告或发布说明已同步限制口径。
4. 主线程明确签收该风险，并接受“限制范围发布”而不是“全量通过”表述。

如果以上任一条件不满足，开放 `P1` 视为阻塞发布。

### 5.3 P2 / P3 门槛

`P2 / P3` 可以不阻塞发布，但必须满足以下条件：

1. 每条问题都有责任人、处理状态和下次复测时间。
2. 已确认不影响最小覆盖计划中的主链路结论。
3. 已在收敛日报或结论中明确写出影响范围和回避方式。
4. 不得把“尚未复测”的问题直接标成“接受风险”。

## 6. 缺陷流转要求

| 状态 | 进入条件 | 下一步 |
|------|----------|--------|
| New | 现场首次发现，证据已生成 | 30 分钟内完成分级与归属 |
| Triaged | 分级、责任人、影响面已明确 | 进入修复、解释或补测 |
| Fixing | 对应线程已接手处理 | 记录预计完成时间 |
| Retest | 已有新包、新配置或新说明 | QA 复测并保留结果 |
| Accepted Risk | 明确不修但已完成风险签收 | 只能用于 P2/P3，且要写清边界 |
| Closed | 已复测通过或确认不再处理 | 保留证据与结论 |
| Blocked | 依赖人、设备或环境未就绪 | 记录阻塞原因与解除条件 |

## 7. 缺陷单最少字段

每个 Round 4 问题至少要补齐下列字段，避免出现“知道有问题，但没人能接”：

| 字段 | 要求 |
|------|------|
| 问题编号 | 推荐 `R4-BUG-xxx` |
| 来源 | 真机矩阵 / smoke / 文案复核 / 日报升级 |
| 设备与环境 | 平台、机型、系统版本、运营商、网络制式 |
| 复现步骤 | 能让其他人复现的最短路径 |
| 实际结果 | 客观描述，不写推测 |
| 期望结果 | 对应当前冻结口径 |
| 证据编号 | 截图、录屏、日志路径 |
| 分级 | P0 / P1 / P2 / P3 |
| 责任归属 | Client / Server / QA / 主线程 / 运维 |
| 当前状态 | New / Triaged / Fixing / Retest / Accepted Risk / Closed / Blocked |

## 8. QA 放行结论模板

建议在日报、结论或主线程回报中使用下面的固定口径：

| 结论 | 使用条件 |
|------|----------|
| 建议发布 | 硬门槛全部满足，且无开放 P0 / P1 |
| 有条件发布 | 仅存在已签收的 P2 / P3，限制范围和回避方式已写明 |
| 暂不建议发布 | 存在开放 P1、待确认必测项或 smoke 阻塞项 |
| 明确阻塞发布 | 存在 P0、主链路失败或放行前置条件不成立 |
