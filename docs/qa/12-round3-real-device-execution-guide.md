# Round 3 真机验证执行指南

## 1. 目标

本指南用于 Round 3 真机验证的现场执行，重点是把兼容矩阵、异常回退和帮助提示连成一套可落地的测试顺序，避免到了真机现场才临时定义口径。

## 2. 执行前准备

1. 确认设备、SIM、运营商、网络制式和系统版本都已记录。
2. 确认测试号码和被叫侧观察方式已准备好。
3. 确认页面文案、帮助页和回退入口为当前冻结版本。
4. 准备截图或录屏工具，异常和待确认项都要留证据。
5. 如果需要引用历史联调结果，只沿用 Round 2 已确认契约，不重新解释 `logout` 语义。

## 3. 测试顺序

建议按下面顺序执行，先易后难，先基线后异常：

1. 设备信息确认：平台、机型、系统版本、运营商、网络制式。
2. 普通拨号基线：不加前缀，确认能否正常拉起系统拨号器。
3. 隐私拨号基线：使用当前冻结前缀，确认拨号器拉起和被叫显示。
4. 拨号方式覆盖：手动输入、联系人、最近号码、复制粘贴逐项覆盖。
5. 异常回退验证：非法号码、设备不支持、拨号器拉起失败、用户取消。
6. 文案与帮助验证：帮助页、能力边界提示、错误提示是否一致。
7. 矩阵回填：把每次尝试的结果写回兼容矩阵和现场记录模板。

## 4. 记录要求

1. 每条记录至少包含时间、设备标识、平台、机型、系统版本、运营商、网络制式、前缀、拨号方式、被叫显示结果、是否成功拉起拨号器、结论。
2. 每次失败都要写“失败点”和“下一步分流方向”，不能只写“失败”。
3. 如果结果看不清，被叫侧显示或拨号器状态必须标为“待确认”，并补证据。
4. 不要把“用户取消”写成系统异常，也不要把“能力边界”写成拨号失败。
5. 当某条记录涉及 Round 2 的 auth 语义时，只能沿用“logout = 注销当前 refresh token”的口径。

## 5. 失败分流方式

| 失败类别 | 先看什么 | 归类建议 | 后续动作 |
|----------|----------|----------|----------|
| 号码非法 | 是否在拨号前就被拦截 | 客户端校验 / 文案问题 | 记录提示文案、输入位置、是否阻止继续拨号 |
| 设备不支持 | 是否为特定机型或系统限制 | 设备能力边界 / 兼容性问题 | 记录机型、系统、运营商、是否需要帮助页提示 |
| 拨号器拉起失败 | 是否连系统拨号器都没打开 | 客户端或系统集成问题 | 记录触发路径、错误提示、截图或录屏 |
| 用户取消 | 是否是用户主动退出 | 正常回退分支 | 记录是否回到原页面、是否保留输入 |
| 帮助提示不一致 | 页面文案是否覆盖当前边界 | 文案 / 帮助页问题 | 记录差异点，回到文案修订清单 |

## 6. 推荐输出

现场执行结束后，建议输出三份东西：

1. 已回填的兼容矩阵。
2. 当日现场记录 / 日报。
3. 需要继续跟进的异常清单。

## 7. 现场判定原则

1. 先判断“能不能拉起拨号器”。
2. 再判断“被叫侧到底显示了什么”。
3. 最后判断“这是设备差异、文案问题还是缺陷”。
4. 不做跨设备的主观推断，必须基于实际记录下结论。
