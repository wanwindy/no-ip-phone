# 联调测试数据与环境准备清单

## 1. 目的

本清单用于 Round 2 联调前的准备确认，保证 `send-code -> login -> me -> dial-prefixes -> notices -> refresh -> logout` 可直接执行。

## 2. 服务与环境

| 项目 | 要求 | 责任归属 |
|------|------|----------|
| 环境类型 | 非生产环境 | 运维 / Server |
| API 地址 | 统一、可访问、已下发给客户端 | Server / Client |
| DB | 可读写，已完成基础迁移 | Server |
| Redis | 可连通，若限流未落地需明确状态 | Server |
| SMS_PROVIDER | `noop` | Server |
| AUTH_FIXED_CODE | `123456` | Server |
| 日志级别 | 可定位联调问题，不暴露敏感信息 | Server |

## 3. 测试账号

| 项目 | 要求 | 责任归属 |
|------|------|----------|
| 手机号 | 至少 2 个可重复使用账号 | Server / QA |
| 验证码 | 统一使用 `123456` | Server |
| 账号状态 | 其中至少 1 个为正常状态，1 个用于异常/封禁验证（如已准备） | Server / QA |
| 设备标识 | 每个客户端实例有稳定 `deviceId` | Client |

## 4. 配置数据

| 数据类型 | 要求 | 备注 |
|----------|------|------|
| dial-prefixes | 至少包含 CN 默认前缀和一个可替换前缀 | 不要求扩展全球化 |
| notices | 至少 1 条有效公告，或明确允许空数组 | 响应结构必须稳定 |
| 默认前缀 | `#31#` | 与方案保持一致 |
| 备注字段 | 说明当前联调环境与限制 | 便于客户端展示 |

## 5. 设备与 SIM

| 项目 | 要求 | 备注 |
|------|------|------|
| Android 设备池 | 至少 1 台 | 仅用于 Round 2 联调，不做兼容结论 |
| iOS 设备池 | 至少 1 台 | 如当前轮不测拨号，仅保留联调客户端验证 |
| SIM | 可拨打或可观察状态的测试卡 | Round 2 不强制做真实通话效果评估 |

## 7. 启动前检查

1. 服务端已使用固定验证码策略启动。
2. 客户端已切换到真实 API 地址。
3. QA 已知晓本轮不做正式发布。
4. 联调记录模板已分发给客户端和服务端参与者。
5. 若限流未落地，相关用例已标记为待验证。
