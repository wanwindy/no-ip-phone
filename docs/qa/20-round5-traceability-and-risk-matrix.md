> 日期：2026-04-22  
> 适用轮次：Round 5 / 第一轮  
> 目标：把 QA 测试项直接映射到任务卡 A/B/C/D，并冻结本轮高风险区与最小证据要求。

## 1. 适用范围

- 本文只覆盖 Round 5 第一轮的多租户语音平台质量基线。
- 本文不替代业务设计，不新增产品能力，只把真源转成 QA 可执行测试项。
- 本文默认遵循以下冻结项：
  - `direct_prefix_mode` 仅为兼容模式
  - 新模式必须走服务端编排抽象
  - 数据隔离采用“共享库 + 行级租户隔离”
  - 回拨命中键为 `display_did + remote_number + TTL`

## 4. 高风险区与失败口径

### 4.1 租户隔离

- 任何跨租户读写、跨租户 DID 可见、跨租户回拨命中，直接判定为 `P0 / No-Go`。
- 若平台级接口与租户级接口边界不清，默认按高风险处理，不允许带风险进入下一轮。

### 4.2 回拨命中

- 命中规则必须严格依赖 `display_did + remote_number + TTL`。
- 若同号回拨可以在错误 DID、错误号码或 TTL 过期条件下成功，直接阻断发布。

### 4.3 DID 外显

- 本轮先验证“展示 DID 字段、映射关系、契约表达”正确，不把真实公网外显效果当作本轮通过前置。
- 若 API、事件、UI 占位中的展示号不一致，按契约阻断处理。

### 4.4 双模式切换

- 新模式必须走服务端编排抽象。
- 旧模式只能作为兼容模式，不得重新成为默认主路径或在文档中被误写成目标架构。

### 4.5 API 契约变更

- `/calls/outbound`、callback 相关字段、event 状态字典、幂等键一旦变化，必须有显式差异记录。
- 无差异记录的结构变更一律不通过发布门禁。
