# 接口联调用例

## 1. 说明

本文件用于阶段 1 的接口联调，覆盖 `auth` 和 `config` 主链路。Round 2 已冻结固定验证码策略，测试前提改为“非生产环境 + `SMS_PROVIDER=noop` + `AUTH_FIXED_CODE=123456`”，且 `send-code` 响应不新增 `debugCode` 字段。

## 2. 通用约定

| 项目 | 约定 |
|------|------|
| 基础路径 | `/api/v1` |
| 响应结构 | `{ code, message, data }` |
| 成功码 | `code = 0`，HTTP 200 |
| 鉴权方式 | 公共接口无需 Bearer；受保护接口使用 Bearer Token |
| 手机号格式 | 中国大陆手机号 `^1[3-9]\d{9}$` |
| 主要错误码 | 40001、40101、40102、40103、40104、40301、42901、42902、42903、50001、50002 |
| Round 2 固定验证码 | 仅在非生产环境且 `SMS_PROVIDER=noop` 时生效，固定值为 `123456` |
| Round 2 响应约束 | `POST /auth/send-code` 仅返回 `cooldown`，不新增 `debugCode` 字段 |
| Round 2 范围 | 仅做 `auth/config` 联调闭环，不展开正式发布动作 |

## 2.1 Round 2 执行前置条件

| 条件 | 说明 | 通过标准 |
|------|------|----------|
| 环境模式 | 非生产环境 | `NODE_ENV != production` |
| 短信提供者 | noop | `SMS_PROVIDER=noop` |
| 固定验证码 | 123456 | 服务端按固定验证码策略可登录 |
| 客户端基础配置 | API base URL 已指向联调环境 | 客户端能完成真实 HTTP 调用 |
| 联调数据 | prefix/notice 数据已预置 | `dial-prefixes` 和 `notices` 可返回稳定数据 |

## 3. 用例清单

### 3.1 `POST /auth/send-code`

| 用例 ID | 场景 | 前置条件 | 步骤 | 预期结果 |
|---------|------|----------|------|----------|
| API-001 | 合法手机号发送验证码 | 满足 Round 2 前置条件 | 提交 `13800138000` | `200`，`code=0`，返回 `cooldown=60` |
| API-002 | 手机号格式非法 | 满足 Round 2 前置条件 | 提交 `123456` | `400`，`code=40001` |
| API-003 | 60 秒内重复发送 | 限流实现已落地 | 再次请求 | `429`，提示发送频繁 |
| API-004 | 1 小时内超限 | 限流实现已落地 | 再次请求 | `429`，提示达到上限 |
| API-005 | 短信供应商失败 | `SMS_PROVIDER` 非 `noop` 且已接入失败模拟 | 请求发送 | `500`，`code=50001` |

### 3.2 `POST /auth/login`

| 用例 ID | 场景 | 前置条件 | 步骤 | 预期结果 |
|---------|------|----------|------|----------|
| API-006 | 正确验证码登录 | 已收到固定验证码 `123456` 或有效验证码 | 提交手机号、验证码、deviceId | `200`，返回 `accessToken`、`refreshToken`、`expiresIn` |
| API-007 | 错误验证码登录 | 已发送验证码 | 提交错误验证码 | `401`，`code=40101` |
| API-008 | 验证码过期 | 验证码已过期或已失效 | 提交过期验证码 | `401`，`code=40102` |
| API-009 | 错误次数过多 | 错误次数达到阈值，且限流/锁定已实现 | 再次提交错误验证码 | `429`，`code=42902` |
| API-010 | 被封禁账户登录 | 账号处于封禁状态 | 提交正确验证码 | `403`，`code=40301` |

### 3.3 `POST /auth/refresh`

| 用例 ID | 场景 | 前置条件 | 步骤 | 预期结果 |
|---------|------|----------|------|----------|
| API-011 | 使用有效刷新令牌刷新 | 已登录 | 提交 `refreshToken + deviceId` | `200`，返回新 token 对 |
| API-012 | 刷新令牌无效 | 无效 token | 提交刷新请求 | `401`，`code=40104` |
| API-013 | 刷新令牌过期 | 过期 token | 提交刷新请求 | `401`，`code=40104` |
| API-014 | 刷新后旧 token 失效 | 已完成一次刷新 | 复用旧 token 再刷新 | 旧 token 应被拒绝或按设计失效 |

### 3.4 `POST /auth/logout`

| 用例 ID | 场景 | 前置条件 | 步骤 | 预期结果 |
|---------|------|----------|------|----------|
| API-015 | 正常退出登录 | 已登录 | 以 Bearer 调用并提交 refreshToken | `200`，服务端注销当前 refresh token |
| API-016 | 退出后复用已注销 refresh token | 已退出 | 使用原 refreshToken 再次调用 `POST /auth/refresh` | `401`，`code=40104` |

### 3.5 `GET /auth/me`

| 用例 ID | 场景 | 前置条件 | 步骤 | 预期结果 |
|---------|------|----------|------|----------|

## 4. 联调关注点

| 关注点 | 说明 | 责任归属 |
|------|------|----------|
| 错误码一致性 | 客户端需要按设计文档映射错误提示 | Server Worker + Client Worker |
| 鉴权边界 | 公共接口与受保护接口不要混用 | Server Worker |
| 刷新重试 | 401 后是否自动刷新并重试 | Client Worker |
| 配置缓存 | 前缀和公告是否有本地缓存策略 | Client Worker |
| 测试数据 | 登录/验证码/公告数据是否稳定可控 | Server Worker + QA |

## 5. Round 2 特别说明

### 5.1 限流相关用例

`API-003`、`API-004`、`API-009` 在 Round 2 中如果服务端限流或错误次数锁定尚未完全落地，应标记为“Round 2 待验证/受实现影响”，不得人工写成通过。

### 5.2 固定验证码用法

Round 2 直接使用固定验证码 `123456` 完成登录主链路验证。若验证失败，优先检查环境变量是否满足 `NODE_ENV != production`、`SMS_PROVIDER=noop`、`AUTH_FIXED_CODE=123456`。

### 5.3 `logout` 语义说明

Round 2 当前契约是“`logout` 注销当前 refresh token”，而不是立即使已签发 access token 失效。因此：

- `logout` 后，原 refresh token 再次 `refresh` 必须失败。
- `logout` 后，已签发且未过期的 access token 在当前实现下可能仍可访问受保护接口。
- QA 不应把“logout 后 `me` 必须立即 401”作为 Round 2 的通过标准。

## 6. 联调通过条件

1. 所有 API 用例的成功路径通过。
2. 登录、刷新、退出和 `me` 的权限边界正确。
3. 前缀与公告接口在登录态下可稳定返回。
4. 客户端能基于这些接口完成主流程联调，不依赖手工改包。
