# 隐私拨号 App Agent Teams 开发模式

## 1. 目标

本文档用于定义“主 agent + 多子线程 agent”的并行开发模式，确保在不相互踩改、不过早耦合、不过度返工的前提下，推进隐私拨号 App 的 `V1.0` 首发开发。

当前推荐目标不是一次性把全部功能做完，而是先用多线程模式跑通第一轮并行开发：

- 主 agent 负责统一契约、线程分工、技术边界、成果验收与集成。
- 子线程 agent 负责按边界完成各自工作包，并输出可直接交接的结果。
- 第一轮重点是“工程骨架 + 契约冻结 + 质量基线”，不是功能细节打磨。

## 2. 推荐团队拓扑

### 2.1 角色划分

| 角色 | 建议名称 | 核心职责 | 写入所有权 |
|------|----------|----------|------------|
| 主 agent | `Lead / Integrator` | 统一需求口径、冻结契约、分发任务卡、验收子线程结果、做最终集成 | `docs/` 下的协作文档、跨模块共享契约、最终集成改动 |
| 子线程 A | `Client Worker` | Flutter 工程骨架、路由守卫、客户端状态与页面壳 | `client/`、客户端实现说明文档 |
| 子线程 B | `Server Worker` | NestJS 骨架、数据库迁移、公共层、接口骨架 | `server/`、服务端实现说明文档 |
| 子线程 C | `QA/Release Worker` | 测试策略、联调清单、兼容矩阵、发布检查表 | `docs/qa/`、发布保障文档 |

### 2.2 第一轮线程目标

| 线程 | 第一轮目标 | 不碰内容 |
|------|------------|----------|
| 主 agent | 固定契约、定义目录边界、收敛接口与阶段目标、整合输出 | 不做大块独立模块开发 |
| Client Worker | 形成可运行 Flutter 骨架与页面/路由闭环 | 不实现服务端与测试资产 |
| Server Worker | 形成可启动 NestJS 骨架与数据库/接口底座 | 不实现客户端与测试资产 |
| QA/Release Worker | 形成测试、联调、兼容、发布准入的执行清单 | 不写业务代码 |

## 3. 共同契约

### 3.1 单一事实来源

所有线程都必须以以下文档为准，冲突时按顺序升级到主 agent 决策：

1. `docs/方案书.md`
2. `docs/设计文档.md`
3. `docs/开发计划文档.md`
4. 本文档 `docs/AgentTeams开发模式.md`

### 3.2 产品与技术硬边界

所有线程必须共同遵守以下不可突破约束：

- 产品定位是“尝试隐藏来电显示”，不是“100% 隐藏号码或归属地”。
- 隐私拨号仅通过运营商前缀加系统电话能力实现，不做云通信中继。
- 客户端必须登录后才能进入拨号主流程。
- 首发版本优先中国大陆场景，默认前缀为 `#31#`。
- 不允许写出误导性文案，例如“必然隐藏”“全运营商可用”。
- 默认不上传拨号历史到服务端。

### 3.3 写入边界

为避免线程冲突，第一轮必须严格遵守目录所有权：

| 线程 | 可写目录 | 禁止直接改动 |
|------|----------|--------------|
| 主 agent | `docs/`、跨线程共享契约文件 | 子线程私有实现目录内的大规模改动 |
| Client Worker | `client/` | `server/`、`docs/qa/` |
| Server Worker | `server/` | `client/`、`docs/qa/` |
| QA/Release Worker | `docs/qa/` | `client/`、`server/` |

如果代码库尚未存在对应目录，子线程可以创建自己的所有权目录，但不能顺手创建别人的工作根目录并填充内容。

### 3.4 升级到主 agent 决策的事项

以下事项不允许子线程自行拍板，必须升级到主 agent：

- 接口字段、响应码、鉴权方式发生变更。
- 数据模型字段与索引发生结构性变化。
- 产品能力表述从“尝试隐藏”偏移到“保证隐藏”。
- 技术栈变更，例如 Flutter/NestJS/数据库/状态管理替换。
- 首发范围调整，例如将 P1/P2 功能提前纳入 P0。
- 子线程需要跨所有权目录修改文件。

### 3.5 回报格式

每个子线程每轮结束必须按同一格式回报：

1. 本轮完成内容
2. 新增或修改文件清单
3. 关键决策与假设
4. 阻塞项与需要主 agent 决策的点
5. 建议下一轮任务

### 3.6 Definition of Done

子线程任务只有同时满足以下条件才算完成：

- 输出物在自己的职责范围内完整闭环。
- 没有跨线程抢写别人的目录。
- 关键假设已经显式写出。
- 交付物能直接被主 agent 集成或继续分发。
- 没有把需要主 agent 决策的问题偷偷固化进实现。

## 4. 主 agent 工作节奏

### 4.1 Round 0

- 固定产品边界
- 固定技术栈
- 固定线程所有权
- 下发第一轮任务卡

### 4.2 Round 1

- 客户端线程建立 Flutter 骨架
- 服务端线程建立 NestJS 与数据库骨架
- QA 线程建立测试与发布保障基线
- 主 agent 做契约核对，不做大块实现

### 4.3 Round 2

- 主 agent 汇总第一轮结果
- 冻结联调接口
- 拆出第二轮“登录联调”“配置联调”“拨号闭环”任务

## 5. 任务卡模板

下面模板由主 agent 统一下发：

```md
# 任务卡：<任务名称>

## 1. 任务目标
- 用 3-5 条描述本轮必须完成的目标。

## 2. 输入上下文
- 引用的设计文档、方案书、开发计划或主 agent 决策。

## 3. 输出物
- 本轮必须交付的代码、文档、脚本或清单。

## 4. 边界与不做项
- 明确哪些事情不属于本任务。

## 5. 写入所有权
- 允许修改的目录或文件范围。

## 6. 完成定义
- 用可以验证的标准定义“完成”。

## 7. 回报格式
- 本轮完成内容
- 文件清单
- 假设与决策
- 阻塞项
- 下一轮建议
```

## 6. 第一轮任务卡

### 6.1 子线程 A：Flutter 客户端骨架

#### 任务目标

- 建立 Flutter 客户端可运行骨架。
- 建立启动页、登录页、验证码页、主页、设置页、帮助页、关于页的基础页面壳。
- 建立 `go_router` 路由与登录态守卫。
- 建立客户端会话状态、本地存储抽象、拨号域模型与拨号封装。
- 为后续后端接入预留 repository / api / provider 接口层。

#### 输出物

- `client/` 工程骨架
- 路由、主题、共享组件
- 本地存储与登录态抽象
- 拨号核心工具与占位接口
- 客户端实现说明

#### 边界与不做项

- 不实现真实短信发送、JWT 服务端逻辑、数据库逻辑。
- 不编写测试矩阵、发布清单、服务端部署。

#### 完成定义

- `flutter run` 可启动。
- 未登录进入登录页，登录态存在时可进入主页骨架。
- 号码输入、普通拨打、隐私拨打入口与前缀配置入口已预留。
- 明确哪些部分是 mock，等待后端接入。

### 6.2 子线程 B：NestJS 服务端骨架

#### 任务目标

- 建立 NestJS 服务端工程骨架。
- 建立数据库实体与迁移，覆盖 `users`、`auth_codes`、`refresh_tokens`、`dial_prefix_configs`、`notices`。
- 建立公共层骨架：统一响应、异常、`@Public()`、JWT strategy/guard、手机号校验 pipe。
- 建立 `auth`、`config` 模块 controller/service/dto 的最小可编译实现。
- 形成后续登录与配置联调的接口底座。

#### 输出物

- `server/` 工程骨架
- `.env.example`
- 数据库迁移与实体
- 公共基础设施骨架
- `auth/config` 模块空壳
- 服务端实现说明

#### 边界与不做项

- 不实现 Flutter 客户端与 UI。
- 不承担测试线程的测试计划、兼容矩阵与发布保障文档。

#### 完成定义

- 服务端可启动。
- 迁移可执行。
- 核心表结构与设计文档一致。
- 接口骨架可供下一轮联调继续接入。

### 6.3 子线程 C：QA / 集成 / 发布保障

#### 任务目标

- 建立阶段 1 的测试策略与需求追踪矩阵。
- 建立接口联调用例、冒烟清单、回归清单。
- 建立机型/系统/运营商/前缀兼容矩阵模板。
- 建立预生产验收与发布检查表。
- 建立风险、阻塞、回滚预案清单。

#### 输出物

- `docs/qa/` 下的测试策略
- 联调用例清单
- 兼容矩阵模板
- 发布检查表
- 风险与阻塞清单

#### 边界与不做项

- 不写客户端或服务端业务实现。
- 不替代主 agent 修改产品能力边界。

#### 完成定义

- 所有交付物都可直接用于测试、联调或发布评审。
- 所有测试点可追溯到现有需求与设计文档。
- 风险与依赖项已标明责任归属。

## 7. 子线程第一轮完整提示词

### 7.1 Client Worker Prompt

```text
你是“隐私拨号 App”的 Flutter 子线程 agent，只负责客户端阶段 1 的最小可并行启动任务。

你不是唯一在代码库里工作的人。不要回滚或覆盖其他线程的改动；只在你拥有的目录里工作，并让你的实现可以被后续线程继续接手。

请严格遵守以下目标与边界：
- 只做 Flutter 客户端，不要实现后端、数据库、短信、风控、部署、CI/CD、测试线程职责。
- 你的任务是把 Flutter 工程推进到“可运行骨架 + 路由守卫 + 客户端会话抽象 + 拨号域封装 + 基础页面壳子”的状态。
- MVP 规则：普通拨打、隐私拨打、前缀设置、帮助说明页、登录态守卫要能在客户端骨架里串起来。
- 隐私拨号仅做系统电话 App 调起，不做静默拨号。
- 默认隐私前缀按 `#31#` 处理，但要保留可配置入口。
- 不要和后端线程抢工作：不要写真实 API 实现、不要写短信发送逻辑、不要写服务端数据模型。
- 不要和测试线程重叠：不要编写测试用例、测试报告、兼容矩阵。

请先按下面 4 个方向组织你的工作，并按优先级推进：
1. 项目骨架与路由：`main/app/router/theme`，`go_router`，启动页，登录态跳转规则。
2. 会话与本地存储：`flutter_secure_storage`、`shared_preferences`、登录态模型、配置缓存抽象。
3. 拨号域与号码工具：号码清洗、格式校验、前缀拼接、`url_launcher` 外部拨号封装。
4. 页面壳与共享组件：登录页、验证码页、主页、设置页、帮助页、关于页、公共按钮/表单/加载态。

实现要求：
- 先搭建清晰目录结构，再补功能。
- 所有后端依赖都用接口、占位实现或 mock，不要硬编码成最终服务端逻辑。
- 登录页和验证码页先做可交互骨架，按钮状态、输入校验、错误态要有。
- 主页要预留号码输入、普通拨打、隐私拨打入口。
- 设置页要预留默认前缀、提示开关、退出登录入口。
- 帮助页要展示“效果受运营商和终端影响”的边界说明。

输出要求：
- 本轮完成内容
- 新增/修改文件清单
- 关键设计决策
- 明确的未完成项与等待后端接入点
- 下一轮建议
```

### 7.2 Server Worker Prompt

```text
你现在是“隐私拨号 App”的 NestJS/数据库 子线程 agent，只负责后端与数据库，不做客户端、测试、产品文案或运维排障。

你不是唯一在代码库里工作的人。不要回滚或覆盖其他线程的改动；只在你拥有的目录里工作，并让你的结果可以被主 agent 直接集成。

请基于以下背景完成第一轮“最小可并行启动任务”，目标是为阶段 1 的核心链路提供可落地的后端底座。

背景：
- 产品是隐私拨号 App，登录后才能使用拨号功能，隐私拨号通过 CLIR 前缀（默认 #31#）尝试隐藏来电显示。
- 阶段 1 要跑通登录、鉴权、配置和服务端主链路。
- 后端技术栈：NestJS 10 + PostgreSQL 15+ + Redis 7 + JWT。
- 数据表：users, auth_codes, refresh_tokens, dial_prefix_configs, notices。
- API 只需覆盖后端契约：POST /api/v1/auth/send-code, /login, /refresh, /logout, GET /me, GET /config/dial-prefixes, GET /config/notices。

本轮你要做的最小并行任务：
1. 搭 NestJS 工程骨架和模块边界。
2. 设计并落地 PostgreSQL 表结构、索引、注释和迁移。
3. 定义后端公共基础设施和 API 契约草案，让后续 auth/config 实现能直接接上。

请优先完成这些交付物：
- 项目目录结构和必要的启动文件。
- .env.example 和必要的配置入口。
- TypeORM 实体/迁移，至少覆盖 users/auth_codes/refresh_tokens/dial_prefix_configs/notices。
- 公共层：统一响应、异常处理、@Public()、JWT strategy/guard 的基础骨架，phone validation pipe 如需可一起建。
- auth/config 模块的 controller/service/dto 空壳或最小可编译实现。
- 一份简短 README 或设计说明，写清楚表结构、关键索引、接口契约和待后续实现点。

边界：
- 不要写客户端代码、页面、路由、Flutter 逻辑。
- 不要写测试用例、测试计划、E2E 脚本。
- 不要做拨号逻辑、号码格式化、前端适配、运营商兼容验证。
- 不要扩展到后台管理、埋点、监控、CI/CD，除非它们是让本轮后端骨架可运行的最小必要项。
- 不要假设未在文档中出现的业务规则；有歧义就保守处理，并在输出里单独标注假设。

输出要求：
- 本轮完成内容
- 新增/修改文件清单
- 关键设计决策与假设
- 需要主 agent 决策的阻塞点
- 下一轮建议
```

```text
你是“隐私拨号 App”的 QA/集成/发布保障子线程 agent。你的任务不是写功能代码，而是在阶段 1 期间，提前并行准备质量保障、联调保障和发布保障材料，确保后续客户端、后端和发布环节可以直接使用。

你不是唯一在代码库里工作的人。不要修改客户端或服务端实现目录；你的职责是提供可执行的测试、联调和发布资产。

请基于以下原则工作：

请输出以下内容：

格式要求：
- 用 Markdown 输出。
- 每个部分短、直接、可执行。
- 不写实现代码。
- 不替客户端和后端做职责内判断。
- 输出最后补充：本轮完成内容、文件清单、阻塞项、下一轮建议。
```

## 8. 主 agent 下一步建议

主 agent 在发出第一轮任务后，建议按以下顺序运作：

1. 先让三个子线程各自只在自己的所有权目录内产出结果。
2. 第一批结果返回后，主 agent 先验收“目录边界是否被遵守”，再看质量。
3. 先冻结接口、数据模型、页面骨架，再进入真实联调。
4. 第二轮只拆“登录联调”和“配置联调”，不要过早展开运营功能。
5. 当客户端与服务端都具备最小骨架后，再新增“集成修复线程”。
