## 2. 主线程验收顺序

1. 是否继续遵守目录边界
2. 是否严格围绕 Round 4 的“验证与收口”，而不是继续扩功能
3. 是否为真机执行与预发布放行提供直接可执行入口
4. 是否引入新的跨线程契约风险

## 3. 核心验收点

### 3.1 Client Worker

- 客户端为真机验证提供更清晰的检查或反馈入口
- 风险提示、帮助文案、错误提示口径一致
- 没有超出首发范围新增功能
- `flutter analyze` / `flutter test` 继续通过

### 3.2 Server Worker

- 服务端生产 / 预生产 Redis 口径明确
- 有最小可执行的服务端 smoke 或环境检查入口
- 有回滚 / 故障分流说明
- `npm run build` 继续通过

### 3.3 QA / Release Worker

- 有 Round 4 真机最小覆盖计划
- 有缺陷分级与放行门槛
- 有预发布 smoke 清单与结论模板
- 有真机执行后的问题收敛入口

## 4. 进入发布准备的条件

只有当以下条件满足时，才建议进入发布准备或预生产验证：

- 真机最小覆盖计划已执行或已具备执行条件
- 新发现的 P0 / P1 问题已收敛或明确阻塞
- 服务端生产环境 Redis 策略已冻结
- 预发布 smoke 与回滚口径已冻结
