# Round 1 验收与 Round 2 决策

## 1. Round 1 验收结论

### 1.1 总体结论

Round 1 通过，允许进入 Round 2 联调阶段。

验收依据：

- `client/` 已形成 Flutter 最小可运行骨架，并通过 `flutter analyze` 与 `flutter test`。
- `server/` 已形成 NestJS 最小可编译骨架，并通过 `npm run build`。
- `docs/qa/` 已形成阶段 1 的测试、联调、兼容、发布保障资产。
- 三条线程均遵守了目录所有权，没有跨目录抢写。

### 1.2 主线程复核结果

主线程已复核以下命令通过：

- `flutter analyze`
- `flutter test`
- `npm run build`

### 1.3 Round 1 接受项

- 接受客户端的页面壳、路由守卫、会话与拨号域抽象。
- 接受服务端的数据库迁移、`auth/config` 模块骨架、统一响应与异常公共层。
- 接受 QA 的测试策略、联调用例、兼容矩阵模板与发布检查表。

## 2. 当前阻塞点

### 2.1 已确认阻塞

- 客户端仍使用 `MockAuthApi` 与 `MockConfigApi`，尚未接真实 HTTP 接口。
- 服务端 `SmsService` 仍为 `noop`，缺少可执行的联调验证码方案。
- 服务端尚未补齐验证码限流与错误次数锁定，QA 中部分接口用例还无法闭环。
- 预生产环境、设备池、SIM 资源仍未冻结，兼容矩阵还只能停留在模板层。

### 2.2 不作为 Round 2 硬阻塞的事项

- 真实短信供应商接入。
- 管理后台。
- 运营分析与更细分运营配置。
- 商店上架与正式发布动作。

## 3. Round 2 决策冻结

### 3.1 验证码联调策略

Round 2 统一采用“非生产固定验证码”策略：

- 当 `NODE_ENV != production` 且 `SMS_PROVIDER=noop` 时，服务端允许使用固定验证码。
- Round 2 默认固定验证码为 `123456`。
- 固定验证码通过环境变量控制，建议使用 `AUTH_FIXED_CODE=123456`。
- `POST /api/v1/auth/send-code` 的响应结构不变，仍只返回 `cooldown`，不在响应体中暴露调试验证码。
- QA 与客户端通过联调说明和环境约定知晓固定验证码，不依赖接口额外字段。

### 3.2 短信策略

Round 2 保持 `SmsService=noop`，不接入真实短信供应商。

原因：

- 第二轮目标是先闭环“登录、刷新、退出、配置拉取”的联调主路径。
- 短信供应商接入会引入额外合规、成本和环境依赖，不是当前最短路径。
- 只要固定验证码策略落地，就足以支撑 QA 和客户端完成主链路联调。

### 3.3 客户端联调策略

Round 2 客户端必须从 mock 切换到真实 HTTP API：

- `AuthApi` 改为真实接口调用。
- `ConfigApi` 改为真实接口调用。
- 保留 mock 能力作为开发兜底时，应通过显式开关控制，不能继续作为默认实现。
- 客户端必须支持可配置 `API_BASE_URL`，推荐使用 `--dart-define` 注入。

### 3.4 服务端联调目标

Round 2 服务端必须补齐以下最小闭环能力：

- `send-code/login/refresh/logout/me` 真正可联调。
- `config/dial-prefixes` 与 `config/notices` 可稳定返回。
- 固定验证码策略可用。
- 基础限流能力开始补齐，至少覆盖验证码发送频率和错误次数累计的最小实现。

### 3.5 QA 联调目标

Round 2 QA 必须把模板推进到“可执行联调资产”：

- 更新接口联调前提，写入固定验证码与环境约定。
- 形成一版真正能执行的联调顺序与 smoke 流程。
- 输出 Round 2 联调验收记录模板。

## 4. Round 2 拆分主题

Round 2 只拆以下三条主线：

1. 客户端真实接口接入
2. 服务端登录/配置联调闭环
3. QA 联调执行资产更新

不在 Round 2 展开以下内容：

- 管理后台
- 正式短信供应商
- 运营化配置
- 应用商店上架

## 5. Round 2 完成标准

只有当以下条件满足时，Round 2 才算完成：

- 客户端默认走真实 API，而不是 mock。
- 服务端在开发环境中能用固定验证码完成登录闭环。
- QA 能按文档跑通 `send-code -> login -> me -> config -> refresh -> logout`。
- 主线程拿到清晰的联调阻塞列表，可以进入 Round 3 的拨号真机与异常回退阶段。
