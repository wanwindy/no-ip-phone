# 多租户语音平台 Agent Teams 开发模式

> 日期：2026-04-22  
> 适用范围：`D:\no-ip-phone` 当前代码库  
> 目标：以“主 agent + 多子线程 agent”模式推进多租户境外 DID 语音平台的设计、建模、实现与集成。

## 1. 目标

本文件用于替代旧的“隐私拨号 App / CLIR 前缀”协作约定，明确当前项目在新目标下的并行开发方式：

- 主 agent 负责真源、共同契约、边界冻结、任务编排、结果验收和最终集成。
- 子线程 agent 在明确写入边界内并行交付，不相互踩改，不私自改契约。
- 第一轮重点不是把语音平台一次性做完，而是先把“多租户基础模型 + 平台边界 + 客户端迁移点 + 语音编排接口面”建立起来。

## 2. 真源

### 2.1 真源优先级

当不同文档或实现之间出现冲突时，按以下顺序判定：

1. `docs/多租户境外号码语音平台设计方案.md`
2. 本文档 `docs/AgentTeams多租户语音平台开发模式.md`
3. `docs/Round5目标与任务拆分.md`
4. `docs/Round5主线程跟踪清单.md`
5. 代码库中的现状实现

### 2.2 真源解释

- 第 1 层是真正的产品和系统目标。
- 第 2 层是协作和边界规则。
- 第 3、4 层是本轮执行面的冻结任务与主线程决策。
- 第 5 层只是“当前事实”，不能反向推翻真源。

## 3. 共同契约

### 3.1 产品边界

所有线程必须共同遵守以下硬边界：

- 当前目标系统是“多租户境外 DID 语音平台”，不是“CLIR 前缀拨号器增强版”。
- 不设计“未知号码/匿名号码/伪造本地号”能力。
- 对外显号的承诺是“固定境外 DID 外显”，不是“所有设备都显示成同一 UI 样式”。
- 回拨能力基于“同号回拨 + 会话 TTL + 目标终端路由”实现。
- 客户端未来主路径是“服务端编排呼叫”，不是继续依赖 `tel:` 直拨。

### 3.2 技术边界

- 客户端主栈继续使用 Flutter。
- 服务端主栈继续使用 NestJS + PostgreSQL + Redis + TypeORM。
- 语音编排层以 `Call Orchestrator + SIP Router + Media Control` 三层建模。
- 第一阶段采用“共享库、行级租户隔离”。
- 旧的 `#31#` 模式仅作为兼容模式保留，不能继续扩为主架构。

### 3.3 决策升级条件

以下事项禁止子线程自行拍板，必须升级到主 agent：

- 表结构或 API 契约发生结构性变化。
- 跨目录所有权写入。
- 角色模型、租户模型、DID 分配模型变更。
- 呼叫状态机、回拨 TTL 规则、计费口径变更。
- Flutter 主流程是否切换到 `server_orchestrated_mode`。
- 是否引入新的基础设施组件。

### 3.4 交付回报格式

每个子线程每轮结束后必须按以下格式回报：

1. 本轮完成内容
2. 新增或修改文件清单
3. 关键设计决策与假设
4. 风险、阻塞与需要主 agent 决策的点
5. 下一轮建议

## 4. 写入所有权

### 4.1 主 agent

- `docs/`
- 跨模块共享契约
- 最终集成改动

### 4.2 子线程建议拓扑

| 线程 | 建议名称 | 核心职责 | 建议写入所有权 |
|------|----------|----------|----------------|
| A | `Tenant & Data Worker` | 多租户、DID、回拨、会话等领域模型与迁移 | `server/src/modules/tenant*`、`server/src/database/migrations`、相关实体 |
| B | `API & App Worker` | 后端应用层接口、鉴权上下文、客户端服务端编排迁移壳 | `server/src/modules/*api*`、`server/src/modules/auth`、`client/lib/features/dialer`、`client/lib/features/config` 中被明确分配的文件 |
| C | `Telephony Contract Worker` | trunk/webhook/event/call-state 契约与接入说明 | `docs/telephony/` 或主线程指定的文档文件 |
| D | `QA & Release Worker` | 质量基线、联调清单、回归/发布准入 | `docs/qa/` |

### 4.3 所有权补充规则

- 如需新增目录，必须在自己拥有的根目录下创建。
- 文档型子线程不得顺手修改业务代码。
- 代码型子线程不得在未授权情况下更新主线程跟踪清单。
- 多个子线程不得同时改同一个现有文件，除非主 agent 明确授予串行修改顺序。

## 5. 主线程节奏

### Round 5 / 第一轮

- 冻结真源与协作边界
- 建立多租户基础模型与任务切口
- 明确客户端从 `direct_prefix_mode` 迁移到 `server_orchestrated_mode` 的抽象层
- 明确 trunk / callback / call events 的契约
- 形成 QA 可执行联调基线

### Round 5 / 第二轮

- 集成租户模型与鉴权上下文
- 接入出呼任务 API 骨架
- 接入 DID/number pool 基础查询与选择
- 完成客户端“创建呼叫任务”链路

### Round 5 / 第三轮

- 接入回拨命中逻辑
- 完成状态事件回流
- 加强审计、报表和账单基础

## 6. Definition of Done

子线程任务必须同时满足以下条件才算完成：

- 只在授权边界内写入。
- 输出物与真源一致，没有把旧的 CLIR 假设重新固化进实现。
- 假设、风险和待决策点都已显式写出。
- 结果能被主 agent 直接集成或发起下一轮。
- 没有通过“先写死、后解释”的方式绕开契约。

## 7. 任务卡模板

```md
# 任务卡：<名称>

## 1. 任务目标
- 本轮必须完成的 3-5 个结果

## 2. 输入真源
- 引用哪些文档、代码现状和主线程决策

## 3. 输出物
- 代码、文档、迁移、接口说明、测试清单等

## 4. 边界与不做项
- 本任务明确不处理的内容

## 5. 写入所有权
- 明确允许修改的目录/文件

## 6. 完成定义
- 可验证的完成标准

## 7. 回报格式
- 本轮完成内容
- 文件清单
- 假设与决策
- 阻塞项
- 下一轮建议
```
