# Round 4 目标与任务拆分

## 1. Round 4 定位

Round 4 进入“真机验证与预发布收口”阶段。

本轮不再扩展新的产品功能，也不再重新打开 Round 1 到 Round 3 已经冻结的核心范围，而是把重点切到：

1. 真机兼容矩阵执行与问题收敛准备
2. 预发布 smoke、放行门槛与回滚口径冻结
3. 客户端 / 服务端 / QA 三侧为上线前验证提供直接可执行资产

## 2. Round 4 共识

- Round 3 已完成客户端拨号体验强化、服务端 Redis 限流收尾与 QA 执行资产准备。
- 本轮优先解决“怎么验证、怎么放行、出了问题怎么回退”，不是继续做新能力。
- 本轮允许围绕真机兼容、错误提示、诊断信息、发布保障做必要补强，但不允许把首发范围继续外扩。

## 3. Round 4 决策冻结

### 3.1 客户端目标

客户端本轮只做“真机验证与预发布支持”，重点包括：

- 围绕真机验证补齐必要的可见信息与问题反馈入口
- 收敛已知风险提示、帮助文案、错误提示的一致性
- 为真机实测和预发布 smoke 提供更直接的检查入口
- 不新增与首发范围无关的功能

### 3.2 服务端目标

服务端本轮只做“预发布与环境收口”，重点包括：

- 冻结 Redis 生产口径与环境变量策略
- 明确预生产 / 生产的启动与 smoke 入口
- 收敛服务端放行条件、最小回滚条件和环境检查项
- 不新增新的业务模块

### 3.3 QA 目标

QA / Release 本轮重点从“资产准备”切到“执行与放行口径”：

- 把真机兼容矩阵拆成最小覆盖计划
- 明确 P0 / P1 / P2 缺陷分级与放行门槛
- 增加 Round 4 预发布 smoke 清单与结论模板
- 输出真机执行后的问题收敛入口

## 4. 线程拆分

| 线程 | 所有权目录 | Round 4 目标 |
|------|------------|--------------|
| Client Worker | `client/` | 客户端真机验证支持、文案一致性、预发布检查入口 |
| Server Worker | `server/` | 预发布配置冻结、Redis 生产策略、服务端 smoke / 回滚保障 |
| QA/Release Worker | `docs/qa/` | 真机最小覆盖计划、缺陷分级、预发布 smoke 与放行模板 |

## 5. 完成标准

### 5.1 Client Worker

- 客户端对真机验证与预发布更友好，但没有引入新的范围膨胀
- 关键帮助文案、风险提示、错误提示口径一致
- `flutter analyze` / `flutter test` 继续通过

### 5.2 Server Worker

- 服务端环境变量、Redis 生产策略、预发布 smoke 入口与放行口径清楚
- `npm run build` 继续通过
- 不把开发态宽松策略误带入生产口径

### 5.3 QA/Release Worker

- 有可执行的 Round 4 真机最小覆盖计划
- 有预发布 smoke 清单和结论模板
- 有缺陷分级、阻塞与放行门槛

## 6. Round 4 不做项

- 不新增管理后台
- 不新增正式短信供应商接入
- 不扩展海外多国家首发范围
- 不做大规模视觉重设计
- 不在没有真机结果的前提下伪造兼容性通过结论
