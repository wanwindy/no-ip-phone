# Round 3 目标与任务拆分

## 1. Round 3 目标

Round 3 进入“拨号体验强化 + 真机验证准备 + Redis 化限流”的阶段。

本轮不再重复做 `auth/config` 基础联调，而是把重点切到：

1. 客户端拨号体验与异常回退优化
2. 服务端限流从内存实现升级为可切 Redis 的可扩展方案
3. QA 将兼容矩阵从模板升级为可执行的真机验证资产

## 2. Round 3 共识

- 本机已经完成 `auth/config` 主链路烟测，通过。
- 本轮不要求在当前机器上完成真实电话呼叫效果验证，因为这需要真机、SIM 卡和运营商环境。
- 但本轮必须把“真机验证前的准备工作”做扎实，让后续只差设备资源，而不是还差代码和文档。

## 3. Round 3 决策冻结

### 3.1 客户端目标

客户端本轮重点不再是接口接入，而是拨号体验：

- 首次隐私拨打风险提示
- 调起失败 / 号码非法 / 配置缺失等错误文案和回退动作
- 最近号码可快速回填
- 帮助页与主页对“能力边界”的提示更清晰
- 为真机验证和用户反馈预留更清楚的交互钩子

### 3.2 服务端目标

服务端本轮重点是 Rate Limit 架构升级：

- 不要求本机必须有 Redis 服务
- 但代码必须支持 `memory` 与 `redis` 两种驱动
- 默认继续兼容 `memory`
- 当未来有 Redis 环境时，应能通过配置切换，不重写 auth 逻辑

建议采用显式驱动配置，例如：

- `RATE_LIMIT_DRIVER=memory`
- `RATE_LIMIT_DRIVER=redis`

### 3.3 QA 目标

QA 本轮重点是真机验证资产，而不是再次扩展 auth/config 用例：

- 把兼容矩阵模板升级为执行版
- 明确机型、系统、运营商、网络制式、前缀、被叫显示结果的采集格式
- 增加拨号异常回退、帮助文案、风险提示的验证项
- 输出 Round 3 现场测试记录模板

## 4. 线程拆分

| 线程 | 所有权目录 | Round 3 目标 |
|------|------------|--------------|
| Client Worker | `client/` | 拨号体验、异常回退、帮助提示、可测性增强 |
| Server Worker | `server/` | Redis 化限流适配层，保留 memory 兼容 |
| QA/Release Worker | `docs/qa/` | 真机兼容验证资产、现场记录模板、Round 3 验收清单 |

## 5. 完成标准

### 5.1 Client Worker

- 拨号相关交互更完整，异常回退更明确
- `flutter analyze` / `flutter test` 继续通过

### 5.2 Server Worker

- Rate limit 架构支持 `memory/redis` 双驱动
- `npm run build` 继续通过
- 文档明确如何切换驱动

### 5.3 QA/Release Worker

- 兼容矩阵从模板变成执行版
- 现场记录表、验证顺序、验收清单可直接拿到真机环境使用

## 6. Round 3 不做项

- 不展开管理后台
- 不接正式短信供应商
- 不做应用商店上架
- 不做大规模 UI 重设计
- 不在没有真机资源的前提下伪造兼容性实测结果
