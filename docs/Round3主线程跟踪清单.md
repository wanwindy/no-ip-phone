# Round 3 主线程跟踪清单

## 1. 当前运行中的子线程

| 线程 | Agent ID | 所有权目录 | Round 3 目标 |
|------|----------|------------|--------------|
| Client Worker | `019db024-eee7-7571-919a-a6ee348b2985` | `client/` | 拨号体验、异常回退、风险提示、真机验证前交互完善 |
| Server Worker | `019db024-ef5d-7823-ba25-a2a71bf534cd` | `server/` | memory/redis 双驱动限流架构 |
| QA/Release Worker | `019db024-ef24-7712-b0af-7c96c2e340bd` | `docs/qa/` | 真机兼容验证资产、现场记录模板、异常回退验证清单 |

## 2. 主线程验收顺序

1. 是否继续遵守目录边界
2. 是否真正围绕 Round 3 目标推进，而不是回退到 Round 2 工作
3. 是否为真机验证与后续收敛提供明确入口
4. 是否引入新的跨线程契约风险

## 3. 核心验收点

### 3.1 Client Worker

- 有首次隐私拨打风险提示与“不再提示”
- 错误提示比当前更细
- 最近号码可回填或再次拨打
- 能力边界和回退路径提示更清楚
- `flutter analyze` / `flutter test` 继续通过

### 3.2 Server Worker

- 有 `RATE_LIMIT_DRIVER`
- 有 memory/redis 双驱动结构
- 默认仍可在无 Redis 环境下工作
- `npm run build` 继续通过
- 文档说明如何切换驱动

### 3.3 QA / Release Worker

- 兼容矩阵升级为执行版
- 有 Round 3 真机验证执行指南
- 有现场记录模板
- 有拨号异常与回退验证清单

## 4. 进入下一轮的条件

只有当以下条件满足时，才建议进入下一轮：

- 客户端拨号体验与异常回退已足够支撑真机测试
- 服务端限流架构已具备 Redis 演进路径
- QA 已可组织真机/运营商矩阵验证
